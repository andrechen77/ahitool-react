# What is it?

Ahitool is a data analytics tool that integrates with JobNimbus to provide
insights on job flows and KPIs. You can visit the tool
[here](https://ahitool.andrechen.org).

# Current features

## Sales KPIs

This tool pulls in every job and related activity from the JobNimbus
database. It computes each job's history of moving through statuses such as
"Lead Acquired" or "Appointment Made", and creates a generalized Sankey diagram
showing the flow of jobs and how long each status transition takes. Users may
add filters to view analytics of only a specific subset of jobs (e.g. to analyze
the effectiveness of a lead source), and customize the definition of the status
groups that define visible transitions on the generated diagram.

# Project structure

The repo is split into a Vite/React frontend at the root and a Cloudflare Worker
that serves the built app and proxies JobNimbus API requests.

```
ahitool-react/
├── src/                      # React frontend
│   ├── pages/                # Route-level pages (Home, Sales KPIs)
│   ├── components/           # Reusable UI (modals, status groups, etc.)
│   ├── contexts/             # React context providers (API key, JobNimbus data)
│   ├── hooks/                # Custom React hooks
│   ├── lib/
│   │   └── job_nimbus/       # JobNimbus API client, KPI logic, IndexedDB cache
│   └── util/                 # Shared utilities
├── deploy_cloudflare/        # Cloudflare Worker deployment
│   ├── src/index.ts          # Worker: API proxy + static asset serving
│   ├── wrangler.jsonc        # Wrangler config (assets, routes, bindings)
│   └── test/                 # Worker tests (Vitest)
├── dist/                     # Production build output (generated by `npm run build`)
├── index.html                # Vite entry HTML
├── vite.config.ts
└── package.json
```

In production, the Worker serves files from `dist/` and forwards
`/api/jobnimbus/*` requests to `app.jobnimbus.com`. In local development, the
frontend runs on Vite (port 5173) and talks to the Worker dev server (port 8787)
via `VITE_API_BASE`.

# Development

## Prerequisites and dependencies

- [Node.js](https://nodejs.org/) (LTS recommended)
- npm

```bash
# Frontend
npm install

# Cloudflare Worker
cd deploy_cloudflare && npm install && cd ..
```

## Environment

Create `.env.development` in the project root so the frontend can reach the local
Worker API proxy:

```
VITE_API_BASE=http://localhost:8787
```

In production builds, `VITE_API_BASE` is omitted and API requests use the same
origin as the deployed site.

## Run locally

Local development uses two processes: Vite for the React app and Wrangler for
the API proxy and asset serving.

**Terminal 1 — frontend (http://localhost:5173):**

```bash
npm run dev
```

**Terminal 2 — Cloudflare Worker (http://localhost:8787):**

```bash
cd deploy_cloudflare
npm run dev
```

Open http://localhost:5173 in your browser. JobNimbus API calls are proxied
through the Worker to avoid CORS issues.

## Other commands

| Command | Description |
|---------|-------------|
| `npm run build` | Type-check and build the frontend to `dist/` |
| `npm run preview` | Preview the production build locally |
| `npm run lint` | Run ESLint |
| `cd deploy_cloudflare && npm test` | Run Worker tests |

# Deploy to Cloudflare

Deployment bundles the Vite build output with the Cloudflare Worker. You need a
[Cloudflare account](https://dash.cloudflare.com/) with Wrangler authenticated
(`npx wrangler login` from `deploy_cloudflare/`).

## Build and deploy

From the project root:

```bash
# 1. Build the frontend
npm run build

# 2. Deploy the Worker and static assets
cd deploy_cloudflare
npm run deploy
```

`wrangler.jsonc` points the Worker's asset binding at `../dist`, so the
frontend must be built before deploying. The Worker is configured to serve
`ahitool.andrechen.org` as a custom domain.

## Worker development commands

Run these from `deploy_cloudflare/`:

| Command | Description |
|---------|-------------|
| `npm run dev` | Local Worker dev server |
| `npm run deploy` | Deploy to Cloudflare |
| `npm run cf-typegen` | Regenerate TypeScript types after changing `wrangler.jsonc` bindings |
